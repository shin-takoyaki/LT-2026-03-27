# bad サンプルの目的

この `bad` フォルダは、**「動くが保守しづらい実装」**の例として、
シンプルなチェックアウト機能を実装しています。

## 実現していること

- リクエストで受け取った `product_ids` から商品を取得する
- 商品ごとに税込金額を計算し、明細と合計金額を作る
- 注文 (`orders` テーブル) を保存する
- 注文結果（注文ID / 合計 / 明細）を JSON で返す

## 処理の流れ

1. `CheckoutController` が HTTP リクエストを受け取る
2. `CheckoutUseCase` が商品取得・金額計算・注文保存をまとめて行う
3. `Order` モデルで `orders` テーブルへ書き込む
4. レシート形式の配列をレスポンスとして返す

## この実装を「bad」としている理由（3つ）

- 依存の向きが外側へ向く（UseCaseがEloquent/DBに直依存）
- テストがDB前提になり、単体テストが難しい
- カラム名や税率ルールの変更に弱い

## 学習ポイント

- 「まず動かす」実装は学習の第一歩として有効
- ただし責務分離（計算ロジック、永続化、入出力）をしないと保守性が下がる
- 次の改善としては、Repository / Domain Service / DTO などへの分割が有効
